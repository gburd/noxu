# Noxu DB v1.5 / v1.6 / v2.0 — Architectural Decisions (2026-05)

> Status: **Awaiting owner sign-off.**
> Source: read-only review of the May 2026 public-API audit
> (`docs/src/internal/api-audit-2026-05-overview.md` and the six
> per-subsystem reports), plus first-hand reading of the production
> source on `fix/sprint1-open-cursor-thread-txn`.

## 1. Purpose and how to use this document

Sprints 1 and 2 cleared the cheapest critical findings:
`Database::open_cursor(Some(&txn), …)` (C1) and
`SecondaryDatabase::open_cursor(Some(&txn), …)` (F4) now thread the
txn through (commits `31d2013` and `8fbb549`). Sprints 3 and 4 are
blocked on three architectural decisions that touch user-visible API
shape and on-disk semantics. This document lays out each decision
with options, rough cost, risk, dependencies, and a recommended
default.

**How to use it.** Read each decision section. Skim §6 for the
recommended-defaults table. Reply "approve" if the defaults are
acceptable, or annotate the table with your preferred option per row.
Once signed off, the table becomes the authoritative roadmap input
for Sprints 3 and 4.

**Honesty about LOC.** All LOC numbers below are *rough order-of-
magnitude*, derived from reading the touched files in the current
tree, not from a careful diff of a reference port. Treat them as
"a few hundred", "a thousand", "several thousand" rather than
precise estimates.

**What is *not* covered here.** Replication audit (out of scope for
the May 2026 pass), XA durability (C5 — separate decision tied to
Phase 7b), DPL atomicity (C6 — separate Persist/XA decision), the
auto-commit isolation hole (F12), and the `EnvironmentConfig`
"config sink" cleanup pass (Theme 1 in the overview). Those should
be tracked separately.

## 2. Decision 1 — Secondary database model

### 2.1 The question

Should Noxu DB's secondary databases follow the **BDB-JE
sorted-dup-key model** (many primaries can map to one secondary
key, stored as duplicates inside the inner index) or commit to a
**one-to-one model** (one primary per secondary key, the inner index
is a non-dup map and any collision overwrites)?

The current implementation is structurally one-to-one. The published
mdBook claims the BDB-JE shape. The two have to converge.

### 2.2 Audit findings driving this decision

* **C2 / F1** — `ForeignKeyDeleteAction`, `foreign_key_database`,
  and the two nullifier traits are stored on `SecondaryConfig`
  (`crates/noxu-db/src/secondary_config.rs:64-110, 174-202`) but
  read by no production code outside that file.
* **C3 / F2** — there is no `associate()` hook. `Database::put` /
  `delete` are unaware of secondaries
  (`crates/noxu-db/src/database.rs:396-489, 586-636`). The rustdoc
  claim "On every primary `put` the secondary is updated via
  `update_secondary`" (`secondary_database.rs:15`) is aspirational;
  `Database::put_and_update_secondaries` does not exist.
* **C4 / F3** — `insert_sec_key` writes the inner entry with
  `Put::Overwrite` (`secondary_database.rs:451-468`). Two
  primaries that produce the same secondary key collide silently.
  The `examples/secondary.rs` example demonstrates this.
* **F4** — *resolved on this branch* (Sprint 1) — secondary
  `open_cursor` now threads the txn. The downstream maintenance
  helpers (`update_secondary`, `insert_sec_key`, `delete_sec_key`,
  `populate_*`) still ignore the txn, so transactional secondary
  *writes* remain only partially atomic with the primary write.
* **F9** — `SecondaryConfig::with_sorted_duplicates(true)` writes
  to `self.base` but `base` is never used to open the inner DB
  (`secondary_database.rs:81-104`). It is a no-op today.

The secondary tree is capable of sorted duplicates. Both
`DatabaseConfig::sorted_duplicates`
(`crates/noxu-db/src/database_config.rs:18,162`) and
`DatabaseImpl::open_with_dups`
(`crates/noxu-dbi/src/database_impl.rs:128-145`) exist and are
exercised by `sorted_dup_test.rs`. The gap is at the secondary
wrapper layer, not in the B-tree.

### 2.3 The three options

#### Option A — Ship sorted-dup secondaries in v1.5

Bring the secondary surface up to BDB-JE shape now: open the inner
`Database` with `sorted_duplicates = true` whenever the
`SecondaryConfig` opts in, change `insert_sec_key` to emit a dup,
change `delete_sec_key` to `SearchBoth` on `(sec_key, pri_key)`,
introduce an `associate()`-style registry on `Database` so
`Database::put` / `delete` automatically maintain registered
secondaries inside the user's txn.

* **Rough LOC:** 1,500–2,500 production + 800–1,500 test.
  Touches `secondary_database.rs` (~900 LOC currently),
  `secondary_cursor.rs`, `database.rs` (associate registry +
  put/delete hook), `database_impl.rs` (no-op — already supports
  dups), the four `populate_*` paths, and the join-cursor
  `get_next_dup` walk.
* **Schedule risk:** **High.** The associate-hook design has many
  edges: weak references to avoid leaks on secondary-close-before-
  primary, recursion handling when two secondaries share a primary,
  ordering with auto-commit writes (still bypassing the lock
  manager — Theme 2 / F12), interaction with the pending FK
  decision (§3). Holding v1.5 on this is a real risk.
* **Doc impact:** rewrite `getting-started/secondary-databases.md`
  end-to-end (drop the manual `update_secondary` pattern); fix
  `transactions/secondary-with-txn.md` (mention `with_transactional`
  is on the *inner* `DatabaseConfig`, drop the bogus
  `env.open_secondary` / `SecondaryConfig::with_transactional`
  references — F6); add a JoinCursor chapter (F7) that finally
  works against true dup intersections (F12).
* **Breakage radius:** moderate. User code that today calls
  `secondary.update_secondary(&pk, old, new)` after every primary
  write keeps working (the call becomes redundant), but the
  intended pattern flips. Inner secondary `Database`s opened by
  the user with `sorted_duplicates = false` will need either a
  migration tool or a versioned-format flag — the on-disk layout
  for one-to-one and sorted-dup secondaries is **not** the same.
  This is the blast-radius tail of Option A.
* **Tests required:** sorted-dup secondary collision, atomic-with-
  txn primary+secondary commit, abort rollback of a multi-
  secondary write, `JoinCursor` over real dup sets, dangling-
  primary skip under `LockMode::ReadUncommitted` (F13),
  associate/disassociate with weak references, populate-on-open
  against a non-empty primary, secondary close-before-primary.

#### Option B — Ship sorted-dup secondaries in v1.6

Same end state as Option A. In v1.5, freeze the surface: keep
secondaries one-to-one, document the limitation prominently in the
mdBook, reject `with_sorted_duplicates(true)` on `SecondaryConfig`
with a clear "unsupported in this version" error, and in v1.5
scope only the *correctness* fixes that do not require sorted-dup
infrastructure: thread `txn` through `update_secondary` /
`delete_sec_key` / `insert_sec_key`, expose `Database::put_and_update_secondaries`
or rename the rustdoc claim to match reality (Theme 4), and reject
`SecondaryConfig` shapes the engine cannot honour today. In v1.6,
land the associate-hook + sorted-dup work as one coherent
release.

* **Rough LOC v1.5:** 300–600 production + 300–500 test (the txn-
  threading and reject-unsupported-config work).
* **Rough LOC v1.6:** 1,200–2,000 production + 800–1,500 test
  (the sorted-dup + associate work, minus what v1.5 already did).
* **Schedule risk v1.5:** **Low.** Almost entirely renaming and
  threading existing parameters.
* **Schedule risk v1.6:** Moderate but de-risked by a v1.5 that
  is honest about what works.
* **Doc impact v1.5:** prominent "one-to-one only" callout in
  `secondary-databases.md` and `secondary-with-txn.md`; fix the
  `with_transactional` / `env.open_secondary` invented APIs;
  document the migration path planned for v1.6.
* **Breakage radius v1.5:** zero for current users (the documented
  path was already one-to-one in practice). v1.6 reintroduces the
  on-disk format question (same as Option A).
* **Tests v1.5:** transactional `update_secondary`,
  abort-rolls-back-secondary, `with_sorted_duplicates(true)` is
  rejected with a typed error, manual-update pattern is the only
  documented path.

#### Option C — Commit to one-to-one and rename the API

Accept that Noxu's secondary index is fundamentally a one-to-one
mapping, rename or reshape the public surface so it cannot be
mistaken for the BDB-JE multi-primary index, and remove the
sorted-dup, associate, and FK affordances entirely (or move them
to a separate `IndexView` / `UniqueIndex` type). This is the
minimum-code path.

* **Rough LOC:** 400–800 production (mostly deletions and
  renames) + 200–400 test.
* **Schedule risk:** Low.
* **Doc impact:** rewrite both secondary chapters from scratch;
  remove `JoinCursor` documentation (joins over a one-to-one index
  add little value over a primary lookup); remove the FK fields
  and traits from `SecondaryConfig`.
* **Breakage radius:** **High at the API surface.**
  `SecondaryDatabase`, `SecondaryCursor`, `JoinCursor`,
  `ForeignKey*` types either disappear or get renamed. Users
  porting code from BDB-JE lose the contract they expect.
* **Tests required:** cursor scan of a unique index; rejection of
  configurations that imply many-to-one.

### 2.4 Recommendation: **Option B** — sorted-dup in v1.6

Rationale. Sorted-dup secondaries with an associate hook are the
right end state — every audit finding (C2, C3, C4, F1–F4, F9, F12,
F13) collapses onto the same workstream, and the project goal is
to honour the BDB-JE contract. But shipping them in v1.5 risks
landing a half-tested associate-hook on a release that already
needs to absorb the Sprint 3/4 cleanups for Themes 1 and 2.
Option B lets v1.5 *honestly* describe what works (one-to-one,
manual `update_secondary`, txn-threaded reads) and lets v1.6
absorb the associate-hook + dup-storage + JoinCursor improvements
as a single coherent release, with the on-disk migration story
planned in advance rather than retrofitted.

The audit's own recommendation is the same shape (overview §
"Recommended next steps" #4 — *"do we ship sorted-dup secondaries
or commit to one-to-one and rename"*). Option B is the path that
keeps the BDB-JE contract on the roadmap without putting it on
v1.5's critical path.

### 2.5 Cross-decision dependencies

* Decision 2 (FK actions): `Cascade` and `Nullify` *both* require
  a working secondary-with-txn write path, which Option B
  ships in v1.6 along with the sorted-dup work. Option A would
  unblock `Cascade` / `Nullify` in v1.5; Option C makes them
  meaningless.
* Decision 3 (nested txn): independent.

## 3. Decision 2 — Foreign-key constraints

### 3.1 The question

`ForeignKeyDeleteAction::{Abort, Cascade, Nullify}`,
`foreign_key_database`, `ForeignKeyNullifier`, and
`ForeignMultiKeyNullifier` are accepted by `SecondaryConfig`
builders, validated for internal consistency, and never consulted
again. Should we (A) implement all three actions in v1.5,
(B) implement only `Abort` in v1.5 (smallest surface — reject
deletes that would orphan), or (C) reject FK config in v1.5 with
a typed "unsupported" error and target v1.6 / v2.0?

### 3.2 Audit findings driving this decision

* **C2 / F1** — config fields are dead. Repo-wide search for
  `foreign_key_database`, `foreign_key_delete_action`,
  `foreign_key_nullifier`, `nullify_foreign_key` returns only
  `secondary_config.rs` and its self-tests
  (`api-audit-2026-05-secondary-join.md` §F1).
* **F15** — the `ForeignKeyTest`-equivalents in
  `crates/noxu-collections/tests/collection_tests.rs:883-995`
  *explicitly state* that the engine has no FK enforcement and
  the tests simulate the behaviour with two cooperating
  `StoredMap`s. There is no `noxu-db` test that exercises a real
  FK action.
* **F16** — `SecondaryConfig.foreign_key_database` is stored as
  `Option<*const Database>` with a hand-rolled `unsafe impl Send`.
  If/when FK is implemented, this representation is a
  use-after-free risk waiting for a user who drops the foreign DB
  before the secondary.

### 3.3 The three options

#### Option A — Implement all three actions in v1.5

`Abort`: on `Database::delete(foreign_pk)`, scan back-references
and refuse if any exist. `Cascade`: scan back-references and
delete the referencing primary records (which itself triggers
recursive secondary maintenance — recursion budget needed).
`Nullify`: scan back-references and call the configured
`ForeignKeyNullifier` to rewrite the primary record.

* **Rough LOC:** 800–1,400 production + 600–1,000 test.
  Adds a back-reference scan to `Database::delete`, a registry
  of "secondaries with FK constraints pointing at me" on the
  foreign `Database`, the recursive Cascade walk, the Nullify
  callback dispatch, the `Arc<Database>` redesign of
  `foreign_key_database` to remove the raw pointer (F16).
* **Schedule risk:** **High.** All three actions need to land
  inside the user's txn. `Cascade` requires the associate-hook
  recursion that Decision 1 Option A would unblock — Option A
  here is **only viable if Decision 1 picks Option A** (sorted-
  dup + associate in v1.5).
* **Doc impact:** rewrite `secondary-databases.md` FK section
  with worked examples for all three actions; replace the
  manual-`update_secondary` pattern (Decision 1 dependent).
* **Breakage radius:** moderate. Users who set FK config today
  expecting it to be a no-op will see deletes that previously
  succeeded now abort — but that *is* the requested behaviour;
  silent FK violation is the bug.
* **Tests required:** Abort rejects with a typed error and the
  txn aborts; Cascade walks N levels; Nullify calls the right
  trait; `Cascade × Cascade` recursion terminates on a cycle;
  abort rolls back partial cascade; close-foreign-DB-before-
  secondary returns a typed error rather than UB (F16); FK
  with multi-key creator and `ForeignMultiKeyNullifier`.

#### Option B — `Abort` only in v1.5, defer `Cascade`/`Nullify`

Implement only the simplest action. On any
`SecondaryConfig::foreign_key_delete_action ∈ {Cascade, Nullify}`,
either reject at `SecondaryDatabase::open` with a typed
"unsupported in v1.5" error or honour `Abort` semantics with a
`log::warn!` that the requested action was downgraded.

* **Rough LOC:** 300–600 production + 300–500 test.
* **Schedule risk:** Low–medium. `Abort` does not need the
  associate-hook recursion; it only needs a back-reference scan
  on the foreign `Database::delete`. The scan can live behind
  the existing manual-`update_secondary` discipline as a
  precondition check at the foreign-DB delete site.
* **Doc impact:** document that `Abort` is supported in v1.5 and
  `Cascade` / `Nullify` are reserved for v1.6 / v2.0.
* **Breakage radius:** small. Users who set `Cascade` / `Nullify`
  today get nothing; in v1.5 they will get a typed error. That
  *is* a breaking change, but only for users who set a config
  flag the engine never honoured.
* **Tests required:** Abort rejects with a typed error;
  Cascade/Nullify configs return `NoxuError::Unsupported` at
  open time; `Abort` interacts correctly with the user's txn
  (the failed delete aborts the txn, not just the operation).

#### Option C — Reject FK config in v1.5; full implementation in v1.6 / v2.0

Add a check at `SecondaryDatabase::open` that rejects any
`SecondaryConfig` with a non-default `foreign_key_database` /
`foreign_key_delete_action != Abort` (or the existence of a
nullifier) with `NoxuError::Unsupported`. Keep the types and
builders so future code does not break, but make every FK
configuration a hard error today. Then ship full FK in v1.6
once Decision 1 lands sorted-dup secondaries (because `Cascade`
needs the associate-hook).

* **Rough LOC:** 50–150 production + 100–200 test (mostly the
  rejection check and its tests).
* **Schedule risk:** Trivially low.
* **Doc impact:** mark FK as "v1.6+" in
  `secondary-databases.md`; remove the FK worked example from
  the rustdoc and replace with "see v1.6 release notes".
* **Breakage radius:** identical to Option B — only users who
  set the dead config flags are affected.
* **Tests required:** the rejection test, plus an "FK is
  documented as unsupported" docs guard.

### 3.4 Recommendation: **Option C** — reject in v1.5, full FK in v1.6

Rationale. The audit's repository-wide search shows zero
production-code call sites that *use* the FK fields today (F1, F15)
— the field is genuinely dead. Rejecting at open is honest,
trivially small, and removes the F16 raw-pointer hazard without a
type redesign. The full FK implementation belongs naturally in
the same v1.6 release as Decision 1's sorted-dup work, because
`Cascade` and `Nullify` both depend on the associate-hook. Option
C keeps v1.5 small and ships the full FK story alongside the
secondary model it depends on.

Option B is acceptable if we want a partial v1.5 capability;
Abort-only is a real value increment over "nothing", and the
back-reference scan is a building block for the v1.6 work. The
risk is that a v1.5 with Abort but no Cascade/Nullify reads as
incomplete in release notes and may invite users to depend on
exactly the FK shape we are about to change.

### 3.5 Cross-decision dependencies

* **Decision 1 is a hard prerequisite for Cascade and Nullify.**
  Both actions modify primary records inside the user's txn,
  which requires the associate-hook from Decision 1.
* **Abort** is independent of Decision 1 — it only needs a
  back-reference scan on the foreign DB's delete path.
* **Decision 3** is independent.
* If Decision 1 picks Option A (sorted-dup in v1.5), Decision 2
  Option A becomes feasible in v1.5. With the recommended
  Decision 1 Option B (sorted-dup in v1.6), Decision 2 Options
  A and B for v1.5 become awkward — only Option C and a partial
  Option B remain on the table.

## 4. Decision 3 — Nested transactions

### 4.1 The question

`Environment::begin_transaction(_parent: Option<&Transaction>, …)`
(`crates/noxu-db/src/environment.rs:600-614`) accepts a parent
argument that is silently ignored — the leading underscore makes
this an explicit choice. The published mdBook
(`docs/src/transactions/basics.md:191-193`) tells the user
"Pass `None` for the parent unless you want a nested (child)
transaction." BDB-JE supports nested transactions and Noxu does
not; we have to either implement them, reject them with a clear
error, or remove the parameter.

### 4.2 Audit findings driving this decision

* **F11** (Transaction-Env) — `_parent` is dropped on the floor.
* The nested-txn surface in noxu-txn is **non-existent**.
  `EnvironmentImpl::begin_txn` (`crates/noxu-dbi/src/environment_impl.rs:901`)
  takes no parent; `TxnManager::begin_txn` exposes no parent
  shape; `Txn` (in `crates/noxu-txn/src/txn.rs`) has no parent
  field, no child-locker hierarchy, no parent-on-commit
  promotion, no abort-on-parent-abort propagation.
* Repo-wide grep: zero call sites pass `Some(parent)` to
  `begin_transaction`. The only `Some(...)` first-argument
  usages all live in tests and pass cursor-internal txns to
  data-path methods, never to `begin_transaction`. The
  blast radius for changing the signature is ~zero in
  production user code; only the mdBook prose mentions a parent.

### 4.3 The three options

#### Option A — Implement nested transactions properly in v1.5

Add a `parent: Option<Arc<Locker>>` field to `Txn` and a parent-
aware `begin_txn` to `TxnManager`. On child commit, transfer the
child's locks and undo log to the parent. On child abort, apply
the child's undo and release its locks while leaving the parent
intact. Update the deadlock detector to recognise the parent-
child lock-set edge. Wire `Environment::begin_transaction` to
build a child txn whose locker has the parent's locker as a
parent.

* **Rough LOC:** 2,000–4,000 production + 1,500–2,500 test.
  Nested txn is a cross-cutting feature: txn manager, locker
  hierarchy, undo log handover, deadlock graph, recovery
  semantics, durability semantics (does a child commit fsync,
  or does only the outermost?).
* **Schedule risk:** **Very high.** This is a multi-sprint
  feature. Every existing transactional path needs a re-audit
  to handle a child txn correctly. There is no infrastructure
  to extend; this is a green-field implementation against a
  published BDB-JE contract.
* **Doc impact:** new chapter in `transactions/`; update every
  example that uses `begin_transaction` to clarify when nesting
  is appropriate.
* **Breakage radius:** zero (additive — `None` keeps current
  behaviour; `Some(parent)` becomes meaningful).
* **Tests required:** child commit transfers locks, child abort
  preserves parent, parent abort cascades to children, nested
  3+ levels, deadlock detector across hierarchies, child reads
  see parent's writes, sibling children isolated from each
  other (BDB-JE: yes, but only because nested txns are
  serialised at the parent), interaction with auto-commit
  writes (already F12-broken).

#### Option B — Reject `Some(parent)` in v1.5 with a typed error

Change `Environment::begin_transaction` to return
`NoxuError::Unsupported("nested transactions are not supported in
this version")` when `parent.is_some()`. Keep the signature.
Update the mdBook prose to document the limitation explicitly.

* **Rough LOC:** 5–20 production + 30–60 test.
* **Schedule risk:** Trivially low.
* **Doc impact:** one paragraph in `transactions/basics.md`
  replacing the misleading "pass `None` unless you want a
  nested transaction" sentence; one entry in
  `operations/known-limitations.md`.
* **Breakage radius:** zero (no production caller passes a
  parent today).
* **Tests required:** rejection test; doc-snippet compile-test
  that doesn't pass a parent.

#### Option C — Remove the `parent` parameter entirely

Drop `parent` from `Environment::begin_transaction`. This is a
breaking change to the function signature — every caller becomes
`begin_transaction(config)` instead of
`begin_transaction(None, config)`.

* **Rough LOC:** 100–300 production + 100–300 test (sprawling
  call-site update across crates and examples; mechanical).
* **Schedule risk:** Low. Mechanical refactor; rust-analyzer
  catches every call site.
* **Doc impact:** every example in `transactions/`,
  `getting-started/`, `replication/`, `collections/` and the
  `examples/*.rs` set must drop the leading `None`. Roughly
  60+ doc snippets and example files (per the
  `begin_transaction` grep — 69 hits across the tree).
* **Breakage radius:** **API breaking** for every existing user.
  The signature changes, so SemVer requires a major bump for
  this alone. If we are saving breaking changes for v2.0 this
  decision belongs there.
* **Tests required:** none specific (every existing test
  validates the new signature mechanically); a compile-test
  that the old `begin_transaction(None, …)` shape is rejected.

### 4.4 Recommendation: **Option B for v1.5; Option C for v2.0**

Rationale. Nested transactions are a feature very few BDB users
actually rely on, the implementation cost is multi-sprint, and
there is **no** infrastructure in `noxu-txn` to extend — Option A
is a green-field feature, not a port. Option B costs almost
nothing, removes the silent-misbehaviour bug today, and is
honest with users about what works. The published mdBook
sentence ("Pass `None` for the parent unless you want a nested
transaction") becomes "Pass `None`. Nested transactions are not
supported." That is a one-line doc change and a one-line code
change.

Option C is correct end state — the `parent: Option<&Transaction>`
parameter is API debt — but it is a SemVer-breaking change to
every user of `begin_transaction`. Defer it to v2.0 and bundle
with whatever other API hygiene we collect (e.g.
`SecondaryConfig` setter-only API per F17, `JoinConfig`
expansion per F18, `Database::put_and_update_secondaries` rename
per Theme 4).

Option A is the right call only if we get a credible signal that
users want it. The audit found no production call sites that
even pass `Some(parent)`; defer until demand exists.

### 4.5 Cross-decision dependencies

* Independent of Decisions 1 and 2.

## 5. Cross-decision dependency summary

```text
Decision 1 (secondary model)
    │
    ├─── feeds → Decision 2's Cascade and Nullify actions
    │             (need the associate-hook)
    │
    └─── Abort branch of Decision 2 is independent

Decision 3 (nested txn) is independent of both.
```

Concrete consequence:

* If Decision 1 = **Option B** (sorted-dup in v1.6), then
  Decision 2 in v1.5 is restricted to **Option C** (reject all)
  or a narrowed **Option B** (Abort only). Full FK lands in v1.6.
* If Decision 1 = **Option A** (sorted-dup in v1.5), then
  Decision 2 = Option A (full FK) becomes feasible in v1.5,
  but at substantially higher schedule risk.
* If Decision 1 = **Option C** (commit to one-to-one), then
  Decision 2 collapses — FK loses its semantic ground; remove
  the FK types and traits.
* Decision 3 is unaffected by either.

## 7. Open questions / what was not covered

The May 2026 audit raised more architectural questions than the
three above. The remainder are *not* on this document's critical
path but should be tracked:

* **C5 — XA prepare durability** is broken end-to-end. Either
  implement the prepared-state log record + recovery promotion
  or document XA as in-process only and reject `xa_prepare`
  outside it. Belongs in a separate Phase 7b decision.
* **C6 — DPL atomicity** — `PrimaryIndex::put` always passes
  `None` for the txn. Touches `noxu-persist` and the
  `EntityStore::beginTransaction` shape. Belongs in a separate
  Persist decision.
* **F12 — auto-commit isolation hole** — auto-commit writes
  bypass the lock manager. Either fix (route auto-commit
  through a synthetic short-lived `Txn`) or prominently
  document the limitation in `transactions/basics.md`. Worth a
  decision of its own.
* **Theme 1 — `*Config` "config sink" cleanup** — every config
  field either gets wired in or rejected at construction time.
  Decision: pick a deadline (v1.6? v2.0?) and a per-field
  policy (wire-in vs reject vs deprecate-and-remove).
* **F22 — typed recovery-failure error variant** —

  every failure mode; should expose a typed
  `RecoveryFailure(reason)` so users can branch on it.
* **F1 (this audit, Sequence) — `Sequence::close` is a no-op;
